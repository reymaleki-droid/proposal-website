Migration Paths

If you start with a simpler package, upgrading later is possible — but the path depends on the architectural boundary between WordPress (A/B) and custom code (C/D/E).

From To What Carries Forward What Must Be Rebuilt Additional Effort
A → C Custom + CRM Product data, brand direction, domain Design, codebase, architecture, admin 80–90% of C
A → E Full ecosystem Product data, brand direction, domain Everything else 90–95% of E
B → C Custom + CRM Product data, design language, categories Codebase, architecture, admin, CRM 70–80% of C
B → E Full ecosystem Product data, design language, categories Codebase, all integrations 80–85% of E
C → E Full ecosystem Entire codebase, database, admin, basic CRM Extended CRM, bot, dashboard, analytics 30–40% on top of C

A and B are WordPress-based. C, D, and E are custom-coded. Moving from WordPress to custom is a rebuild, not an upgrade. Moving from C to E is an extension. This is the most important architectural decision.
